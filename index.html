Impolice

Android & WEB & Server



1. 프로젝트 소개

본 프로젝트는 외주를 받아 진행했으며, 사람들의 무분별한 불법 행위를 신고할 수 있도록 도와주는 어플리케이션을 개발했습니다.
어플리케이션에서 영상과 신고정보를 통해 신고를 하면, 관리자 페이지에서 신고 목록들을 확인하며 신고처리 여부를 결정합니다.

2. 프로젝트 팀원 소개

- 개발자 1명
- 디자이너 1명
- PM 1명

3. 프로젝트 주요 기능

- UI 구축 및 Retrofit 통신
- 동영상 편집 기능
- 웹 페이지에서 신고내역 변경에 따른 FCM 알림
- Google Map 3rd Party API 사용
- Paging 기법 사용
- 그래프, 지도, 신고 목록 테이블을 포함한 웹 페이지 구축 및 Ajax 통신
- 신고 목록들 엑셀 다운로드

4. 프로젝트 후기

본 프로젝트에서 동영상 편집 기능이 가장 난이도가 있었습니다. 처음부터 이 기능을 구현하면 시간이 너무 많이 들고 구현이 불확실했습니다. 그래서 상용화할 수 있는 동영상 편집 오픈소스 HGLVideoTrimmer를 찾았습니다. 먼저 오픈소스 전체 흐름을 이해한 다음, 필요한 부분의 event를 catch하고 모듈 안에 필요 기능을 추가하는 방식으로 개발했습니다.

신고 목록들을 보여줄 때 한 번에 많은 양의 데이터를 받아올 경우 네트워크 낭비가 심해 Paging 기법을 사용하였습니다. Recyclerview에서 제일 마지막 아이템을 스크롤을 했을 때 last index를 서버에 보내고, 서버로부터 일부 데이터를 받아 notify하였습니다.

사진을 firebase storage에 저장한 후, DB에 url을 저장하는 방식으로 수행하였습니다. 처음에 서버안의 directory에 사진을 저장하는 방식을 생각했으나 이미지 관리, 서버 용량, 속도 문제로 firebase storage에 저장하는 방식을 채택하였습니다. 이때 Firebase 저장소 용량을 줄이기 위해 클라이언트에서 사진을 압축하고 storage에 업로드 하였습니다.

관리자 페이지에서 신고내역을 변경하면 사용자에게 실시간으로 알리기 위해 FCM을 사용하였습니다. 왜냐하면 웹서버와 어플리케이션은 Http 프로토콜을 사용하므로 서버에서 클라이언트로 Request 없이 데이터를 전송 못하기 때문입니다.

이렇게 개발실력 향상 뿐만이 아니라 팀원들과 개발적으로 소통을 어떻게 해야하는지도 또한 배웠습니다. 우선 팀원들과 소통을 할 때 제플린 및 슬랙이라는 프로그램을 이용하여 디자이너, 서버개발자와 소통을 하였습니다. 이 과정에서 서버개발자는 어떠한 일을하고 디자이너는 무슨일을 하지에 대해서 각자의 역할을 알아가는 시간을 가졌습니다.

AI Kiosk

Android & Server



1. 프로젝트 소개

많은 지역에서 키오스크가 도입되고 있지만 노인분들 포함한 멀티 기기에 익숙하지 않은 분들은 키오스크를 사용하는 것이 불편하다. 그래서 키오스크에서 누적된 데이터를 활용해 메뉴를 추천해주고, 음성 대화를 통한 AI 키오스크를 개발했습니다.

2. 프로젝트 팀원 소개

- 개발자 3명
- 디자이너 1명

3. 프로젝트 주요 기능

- Android에서 제공하는 Camera Face detection을 이용한 사용자 얼굴 실시간 감지
- CFR API를 사용하여 얼굴 특징파악
- CPV API를 사용하여 음성서비스 제공
- CSR API를 통해 사용자의 음성데이터를 text로 추출한다. 이때 좀 더 정확한 데이터를 얻기 위하여 서버 DB에서 후보정 처리 진행
- CFR API에서 얻은 데이터와 사용자 주문 내역을 비교하여 사용자에게 추천메뉴 제공

3. 프로젝트 후기

어플리케이션 시나리오는 사용자가 태블릿 근처로 오면 얼굴인식을 수행 후 해당 뷰를 JPG 변환 후 CFR API를 통해 얼굴 특징을 파악합니다. 이후 사용자 연령, 성별에 맞게 메뉴를 추천 해주고, 주문 내역을 바탕으로 서버에 데이터를 저장합니다.
본 프로젝트에서 Face Detection과 CFR API를 이용하여 얼굴에 대한 특징을 파악하는 기능구현과 음성인식에서의 후보정 처리 기능을 맡았습니다. 얼굴 인식은 Github Opensource로 돼있는 코드를 활용했으며, 음성 인식의 정확도를 높이기 위해 음성 인식 결과를 DB에 축적하는 방식으로 후보정 처리를 진행했습니다.

이런알람

Android



1. 프로젝트 소개

평소 영어를 읽고 쓰는 것은 수월하지만, 외국인과 대화하는 것이 힘든점에 출발하였습니다. 그래서 알람 기능을 통해 영어 발음을 주기적으로 공부할 수 있는 어플리케이션을 출시하였습니다.

2. 프로젝트 팀원 소개

- 안드로이드 2명 (본인은 알람 기능을, 다른 팀원은 음성 인식 개발)

3. 프로젝트 주요 기능

- Alarm Manager, Broadcast Receiver, Service를 이용한 알람 기능 개발
- SQLite를 통해 알람 요일, 시간을 저장하였고 알람 설정 데이터는 Sharedpreference를 이용하여 저장
- 구글에서 제공하는 SpeechRecognizer 클래스를 통해 음성인식을 수행

4. 프로젝트 후기

이번 프로젝트에서 “알람 기능”에 대해 배울 수 있었습니다. 다만, 알람 기능을 테스트하기 위해 최대 1주가 필요하여 불편 했지만, 자주 사용하지 않던 Broadcast Receiver와 Service를 사용할 수 있는 좋은 기회였습니다. 초기에 알람 동작 원리를 이해하는데 어려움을 느꼈고 아래와 같이 동작하도록 개발했습니다.
-1. Pending Intent에 NotificationID와 어떤 Broadcast Receiver를 호출할지를 추가합니다. 이후 이를, Alarm Manager에게 보내고 Alarm Manager는 Set or SetExact 함수를 통해 알람을 등록합니다.
-2. 이제 정해진 시간이 되면 OS에서 Broadcast하여 해당 Notification ID를 가진 Broadcast Receiver가 수신하여 Service를 호출하는 방식으로 진행했습니다.

핸드폰 기종마다 알람 시간이 다른 문제가 발생했습니다. 이는 Android OS 버전마다 지원하는 함수가 달랐기 때문입니다. 저는 이 문제를 핸드폰 API마다 알람 등록 함수를 다르게 설정하여 해결했습니다.

알람을 강제로 끄는 것을 방지하기 위해 안드로이드에 내장된 홈 버튼을 막는 기능을 개발했습니다. 뒤로가기 버튼은 override를 통해 쉽게 막을 수 있었지만 그 외의 버튼은 안드로이드에서 지원하지 않는다는 것을 알았습니다. 그래서 저는 Activity 생명주기를 이용했습니다. Activity가 onstop으로 갈 경우 다시 activity를 호출 했으며, onPause에서는 ActivityManager를 이용했습니다.

이번 프로젝트를 통해 안드로이드 주요 Component, 생명주기, 같은 코드여도 SDK마다 지원하는 함수가 다를 수 있음을 배웠습니다.

4. 프로젝트 후기

Amotag 서버로부터 HR Server를 할당 받았고, Amotag DB로부터 직원 정보를 불러와 HR Server DB에 동기화하는 작업을 수행하였습니다. 이때 Amotag DB로 접근할 수 있는 ID를 할당 받아 정해진 시간마다 API를 호출하여 ERP 동기화 작업을 수행하였습니다

프로젝트에서 어려운 기능은 없었으나 DB를 설계하는 과정에서 많은 노력이 필요했습니다. 법인, 부서, 사원의 관계를 높이가 최대 6인 트리구조 형태로 진행했습니다. 각각의 법인은 루트 노드이며, internal 노드는 부서를 가리키고, 직원은 External 노드로 표현하였습니다. 이때 각 노드의 ParentID 컬럼에 부모 ID를 저장했습니다.
또한, 검색 기능과 직원 정보 다운로드 기능을 개발하면서 생각보다 복잡한 쿼리문을 작성하면서 쿼리를 어떻게 작성해야 되는지 배울 수 있는 기회였습니다.
초기 개발 시 데이터가 적어 엑셀 다운로드가 정상적으로 동작했지만, 약 1000명의 실 데이터를 넣으면서 Timeout 이슈가 발생하였습니다. 물론 Timeout이 발생하지 않도록 쿼리를 다듬으면 좋지만, 한 번에 많은 양의 데이터를 다운로드하는 특성 때문에 이를 Nginx의 timeout 설정 값을 바꾸어 해결했습니다. 또한, 많은 사용자들이 사용하면서 예상치 못한 곳에서 버그가 나왔고, Validation은 필수로 체크해야 됨을 배웠습니다.

프로젝트를 진행하면서 3번의 PM 인수인계가 발생하면서 많은 이슈가 생겼습니다. 고객사와의 마찰, 그리고 개발자와의 마찰 또한 생겼습니다. 이를 통해 인수인계의 중요성을 느꼈고 팀원들과 어떻게 갈등을 해결하면 좋은지에 배웠습니다.
